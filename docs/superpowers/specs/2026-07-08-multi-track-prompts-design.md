# Мульти-трек: редактируемые промпты-персоны

**Дата:** 2026-07-08
**Статус:** одобрено, в реализации

## Проблема

Фрилансер вручную гоняет вакансии через Claude с промптом-персоной (кто он,
его офферы, ставка) и получает вердикт + черновик отклика. Он берёт два вида
работы — разработку и лендинги — с разными скиллами, портфолио и голосом.
Сейчас в приложении один глобальный профиль из `stack_profile.yaml`,
редактируется руками на диске. Нужно: (1) редактируемые из UI персоны,
(2) два трека, (3) каждая вакансия скорится и получает черновик под свою персону.

## Решение

**Трек = редактируемая персона.** Новая модель `Track` держит весь профиль
(персона, промпты, скиллы, ставка, проекты, red-flag-фразы). Сохранённый поиск
(`SavedFilter`) привязан к треку; вакансия наследует трек от поиска, который её
нашёл (`job.matched_filter.track`). Скоринг, письмо и скрининг резолвят трек
вакансии и работают под его персоной. Нет трека → дефолтный трек (фолбэк).

### Архитектура

- **`apps/tracks`** (новое мини-приложение, ни от кого не зависит — чтобы не
  плодить циклы; его импортируют `jobs`, `scoring`, `letters`, `screening`).
  - `Track(TimeStampedModel)`: `name`, `scorer_role`, `job_analysis_prompt`,
    `cover_letter_instructions`, `screening_instructions`, `signoff`,
    `skills` (JSON list), `min_hourly_rate` (int), `projects` (JSON list of
    `{repo, skills}`), `red_flag_phrases` (JSON list), `is_default` (bool).
  - `Track.get_default()` — дефолтный трек (первый `is_default=True`, иначе
    первый по id); используется как фолбэк.
  - Data-миграция сидит трек «Разработка» из `stack_profile.yaml`, `is_default=True`.
- **`SavedFilter.track`** — FK на `Track`, `null=True on_delete=SET_NULL`
  (jobs-миграция). Вакансия знает свой поиск (`matched_filter`), значит и трек.

### Поток данных

- **Резолв** (`apps/scoring/profile.py`):
  - `resolve_track(job) -> Track` — `job.matched_filter.track` или `Track.get_default()`.
  - `track_config(track) -> dict` — единый словарь той же формы, что раньше
    возвращал `load_profile()`, плюс `scorer_role`/инструкции/`signoff`. Его
    читают скорер, llm-скорер, письмо, скрининг без изменений формы.
  - `freelancer_config()` ужимается до идентити для сайдбара
    (`name/display_name/initials/headline`), продолжает читать yaml. Идентити —
    личное и общее, не пер-трек.
- **Скоринг** (`scorer.py`): `score_job(job, *, track=None)` резолвит трек,
  строит `track_config`, дальше `_compute`/`llm_compute` без изменений (та же
  форма профиля). Кэш эмбеддинга профиля ключуется по `track.id`.
  `score_pending_jobs` резолвит трек на каждую вакансию.
- **Письмо** (`letters/generator.py`): `generate_cover(job)` резолвит трек →
  `track_config` для инструкций/подписи и `track.projects` для подбора
  репозиториев. `MockGitHub(projects=...)` берёт портфолио трека; реальный
  `GitHubClient` остаётся глобальным (аккаунт один — известное ограничение).
- **Скрининг** (`screening/generator.py`): `generate_answer(question)` резолвит
  трек из `question.job` для `screening_instructions`.

### Страница настроек

- `/settings/tracks/` — список треков + форма редактирования (Django ModelForm,
  все поля; JSON-поля как textarea, парсятся по строкам/JSON), кнопка «создать».
- `apps/tracks/views.py` + `urls.py` + шаблоны в стиле `base.html`.
- Включается пункт «Настройки» в сайдбаре (`base.html`, сейчас `disabled`).
- Подключить `apps.tracks.urls` в `config/urls.py`.

### Обработка ошибок

- Битый JSON в форме настроек → ошибка валидации формы, не 500.
- Вакансия без трека / поиск без трека → дефолтный трек.
- Нет ни одного трека (свежая Б, до сидинга) → миграция гарантирует один; на
  всякий случай `get_default()` терпит пустоту и код не падает на `None`.

### Тестирование

- Резолв: job→filter→track; фолбэк на дефолт при `matched_filter=None` и
  `track=None`.
- Скорер использует скиллы/ставку/red-flags своего трека (два трека → разный score).
- Письмо использует подпись и проекты своего трека.
- Скрининг использует инструкции трека вакансии.
- Форма настроек: сохранение, отклонение битого JSON.
- Data-миграция создаёт дефолтный трек.

## Вне объёма (следующие шаги)

- «Вопросы клиенту» как отдельный вывод LLM (новое поле в `JobScore` + место в UI).
- Автоклассификация вакансии в трек независимо от поиска.
- Пер-трек портфолио для реального GitHub-аккаунта.
